# Screenshot Captions for Report

## Figure 1. Project Folder Structure

This screenshot shows the final project directory opened in Visual Studio Code. The project is organized into separate folders for the MQTT publisher, MQTT subscriber, Streamlit dashboard, database scripts, Mosquitto configuration, requirements files, screenshots, and report files. This structure makes the system easier to understand, run, and maintain.

## Figure 2. Docker Compose Initialization

This screenshot shows the execution of the `docker compose up -d` command. Docker Compose starts the required services for the project, including the Mosquitto MQTT broker, MySQL database, MongoDB database, and Neo4j graph database. This confirms that the project environment can be created using a single command.

## Figure 3. Docker Services Started Successfully

This screenshot confirms that the Docker containers for Mosquitto, MySQL, MongoDB, and Neo4j were created and started successfully. It verifies that the database and MQTT services were available before running the Python publisher and subscriber applications.

## Figure 4. Python Virtual Environment and Dependency Installation

This screenshot shows the Python virtual environment activated before installing and running the project dependencies. The required packages include the MQTT client library, MySQL connector, MongoDB driver, Neo4j driver, python-dotenv, and Streamlit dashboard dependencies.

## Figure 5. Subscriber Connected to the MQTT Broker

This screenshot shows the MQTT subscriber running from the terminal. The subscriber connects to the Mosquitto broker on `localhost:1883` and subscribes to three MQTT topics: `environmental/network`, `environmental/readings`, and `environmental/telemetry`. This confirms that the subscriber is ready to receive different kinds of IoT messages.

## Figure 6. Publisher Sending Data to Three MQTT Topics

This screenshot shows the sensor publisher running and publishing simulated environmental data. The publisher sends topology messages to `environmental/network`, structured measurement messages to `environmental/readings`, and raw telemetry messages to `environmental/telemetry`. This demonstrates the topic-based design of the system.

## Figure 7. Subscriber Routing Messages to the Correct Database

This screenshot shows the subscriber receiving MQTT messages and routing them based on topic. Readings are written to MySQL, telemetry is written to MongoDB, and network topology is written to Neo4j. The terminal output also shows the write latency for each database operation, which supports the performance evaluation part of the project.

## Figure 8. MySQL Latest Readings

This screenshot shows the latest records stored in the MySQL `readings` table. MySQL stores the structured environmental measurements, including sensor ID, temperature, humidity, air quality, reading time, status, and creation time. This verifies that structured sensor readings are being saved correctly.

## Figure 9. MySQL Readings Grouped by Sensor

This screenshot shows an SQL query that counts the number of readings generated by each sensor. The result confirms that multiple sensors are producing data and that MySQL can be used for structured analysis using SQL aggregation.

## Figure 10. MySQL Alerts with Alert Messages

This screenshot shows an SQL query joining the `alerts` table with the `alert_types` table. The result displays alert IDs, reading IDs, sensor IDs, alert types, human-readable alert messages, and alert times. This verifies that abnormal environmental conditions are classified and stored as alerts.

## Figure 11. MongoDB Shell Connection

This screenshot shows access to the MongoDB container using `mongosh`. It confirms that MongoDB is running correctly and that the project can connect to the database from the command line.

## Figure 12. MongoDB Collections

This screenshot shows the `sensor_messages` collection inside the `env_monitoring` database. This verifies that MongoDB is being used to store raw telemetry messages produced by the MQTT pipeline.

## Figure 13. MongoDB Raw Telemetry Documents

This screenshot shows JSON-like telemetry documents stored in MongoDB. The documents include environmental values such as temperature, humidity, and air quality, together with flexible device metadata such as battery level, firmware version, signal strength, error code, and maintenance notes. This demonstrates why MongoDB is suitable for semi-structured IoT data.

## Figure 14. Neo4j Full Sensor and Location Network

This screenshot shows the complete graph stored in Neo4j. The graph contains Building, Floor, Room, and Sensor nodes connected through relationships such as `IN_BUILDING`, `ON_FLOOR`, and `LOCATED_IN`. This confirms that Neo4j is being used to model the sensor/location topology.

## Figure 15. Neo4j Sensor-to-Room Relationships

This screenshot shows the `LOCATED_IN` relationships between sensors and rooms. Each sensor is connected to its physical room, such as Room A, Room B, or Laboratory. This verifies that Neo4j correctly represents the placement of sensors in the monitored environment.

## Figure 16. Neo4j Node Labels and Relationship Types

This screenshot shows the Neo4j database information panel. It lists the node labels, relationship types, and property keys used in the graph. This confirms that the graph model contains the expected labels and relationships for the sensor/location network.

## Figure 17. Streamlit Dashboard Connection Status

This screenshot shows the Streamlit dashboard successfully connected to MySQL, MongoDB, and Neo4j. The dashboard confirms that all three databases are reachable and that the system can present data from different database platforms in one interface.

## Figure 18. Streamlit System Overview

This screenshot shows the dashboard overview with the number of MySQL readings, MongoDB documents, Neo4j nodes, and Neo4j relationships. These values provide a quick summary of the amount of data stored in each platform during the test run.

## Figure 19. MySQL Readings Displayed in the Dashboard

This screenshot shows the latest MySQL readings displayed in the Streamlit dashboard. The dashboard also adds the sensor location by using the sensor ID and Neo4j location data. This demonstrates a cross-database link between structured relational data and graph-based location data.

## Figure 20. Environmental Values Over Time

This screenshot shows a line chart of temperature, humidity, and air quality values over time. The chart is generated from MySQL readings and demonstrates how the collected IoT data can be analyzed visually.

## Figure 21. MongoDB Telemetry Displayed in the Dashboard

This screenshot shows raw telemetry documents from MongoDB inside the Streamlit dashboard. The flattened columns display both normal sensor values and flexible nested device metadata, showing MongoDB’s advantage for storing variable IoT messages.

## Figure 22. Firmware Metadata from MongoDB

This screenshot shows the dashboard section that analyzes firmware versions reported by devices. The information comes from MongoDB telemetry documents, where device metadata may vary from one message to another.

## Figure 23. Neo4j Network Summary in the Dashboard

This screenshot shows the Neo4j sensor/location network summary in the dashboard. It displays node labels and relationship types as tables and charts, proving that the graph database is successfully integrated into the monitoring interface.

## Figure 24. Dashboard Sensor Placement View

This screenshot shows the sensor placement table generated from Neo4j. It maps each sensor ID to its room, allowing the dashboard to connect MySQL readings with their physical locations.

## Figure 25. End-to-End System Verification

This screenshot summarizes the final working system. The MQTT publisher sends simulated environmental data, the MQTT broker forwards the messages, the Python subscriber processes and routes the data, and the three databases store the correct type of information. The Streamlit dashboard then reads from all databases and displays the final result.
